Objetivo
- Implementar a automação de testes de API REST do sistema Cantina utilizando Mocha, Chai, Supertest e Mochawesome.
- Cobrir todas as User Stories descritas no README.md, transformando cada Condição de Teste do arquivo "4 - Condições de Teste.txt" (camada API) em pelo menos um caso de teste automatizado.

Contexto
- O backend está em Node.js + Express, com persistência em memória, autenticação JWT em middleware e dados zerados a cada inicialização. Ponto de entrada: server.js.
- A especificação OpenAPI está em "resources/swagger.yaml" e descreve todos os endpoints, schemas e códigos de erro.
- As User Stories estão no "README.md" (1 a 15) e seus Critérios de Aceitação Gherkin servem como referência de comportamento esperado.
- As Condições de Teste estão em "Prompts/Planejamento e criação do software/4 - Condições de Teste.txt", organizadas por User Story e numeradas a partir de CT01. A coluna "Camada" indica se o teste é mais apropriado em API (escopo deste prompt) ou Web.
- A API local roda em "http://localhost:3000". Em CI a porta pode variar.
- Existem dois usuários pré-cadastrados no seed (admin e cliente). Para cenários que exijam outros perfis/estados, criar usuários via /auth/register + /auth/confirm-email.

Regras
- Stack obrigatória: mocha, chai, supertest, mochawesome. Pode acrescentar sinon (para fake timers no horário de corte) e dotenv (para variáveis de ambiente).
- Estrutura de pastas (na raiz do backend):
    test/
    ├── api/
    │   ├── us01-login/
    │   │   └── login.spec.js
    │   ├── us02-cadastro-reservas/

    │   ├── us03-cancelamento-cliente/
    │   ├── us04-consulta-reservas/
    │   ├── us05-marcar-entregue/
    │   ├── us06-cancelamento-admin/
    │   ├── us07-cadastro-cliente/
    │   ├── us08-confirmacao-email/
    │   ├── us09-recuperacao-senha/
    │   ├── us10-perfil/
    │   ├── us11-detalhe-reserva/
    │   ├── us12-listagem-global-admin/
    │   ├── us13-historico-cancelamentos/

- Um arquivo .spec.js por User Story; dentro dele, um describe por User Story e um describe aninhado por agrupamento (por exemplo: "Validações", "Regras de negócio", "Permissões"). Cada it deve referenciar a Condição de Teste correspondente no nome (ex.: "CT04 — bloqueia usuário após 5 tentativas inválidas").
- Toda Condição de Teste de camada API do arquivo "4 - Condições de Teste.txt" precisa ter no mínimo um it associado. Casos de teste mais específicos podem ser adicionados quando agregarem valor (ex.: testar quantidade=0, decimal e negativa para a CT04 da US02).
- Usar Data Driven Testing onde se aplicar (validações com múltiplos valores, matrizes de combinações). Usar describe.each / forEach com fixtures em JSON. Não duplicar copy-paste de cenários.
- Hooks:
    - before global: limpar e (re)inicializar o estado da API garantindo que o seed esteja aplicado.
    - beforeEach por bloco: obter token de cliente/admin via helper de auth, criar reservas pré-condição quando necessário.
    - after/afterEach: limpar artefatos criados (cancelar reservas, restaurar fake timers do sinon).
- Para regras dependentes de hora (horário de corte), usar sinon.useFakeTimers para travar a "hora atual" e tornar os testes determinísticos. Não usar sleep nem aguardar a virada de hora real.
- Para verificação de mensagens de erro, ler a mensagem real retornada pela API (campo "message" no payload de erro) e comparar com a mensagem documentada nas regras/Gherkin do README. Quando a mensagem mudar no backend, o teste deve mudar junto — proibido aceitar "qualquer string".
- Cada it deve assertar:
    - Status code esperado (chai expect(res.status).to.equal(...)).
    - Estrutura do payload de sucesso/erro (chai expect(res.body).to.have.property(...) ou .to.deep.include(...)).
    - Conteúdo das mensagens de erro quando aplicável.
    - Side effects relevantes via outra chamada (ex.: após criar reserva, GET /reservations/availability deve refletir o novo "used").
- Configuração:
    - Usar dotenv para "BASE_URL" (default http://localhost:3000), "ADMIN_EMAIL", "ADMIN_PASSWORD", "CLIENT_EMAIL", "CLIENT_PASSWORD" e demais credenciais.
    - Não chumbar URLs nem credenciais nos specs.
    - O helper http.js deve criar o agente supertest a partir de BASE_URL.
    - Para a API ser testada com .env próprio (cortes/estoques) sem afetar produção, considerar um arquivo "test/.env.test" carregado pelo helper.
- Comandos npm (em package.json):
    - "test": "mocha --config test/.mocharc.cjs 'test/**/*.spec.js'"
    - "test:report": "mocha --config test/.mocharc.cjs --reporter mochawesome --reporter-options reportDir=test/reports,reportFilename=index 'test/**/*.spec.js'"
    - "test:us": npm run test -- --grep "USXX" (para rodar uma US específica)
- Cobertura mínima por User Story:
    - 1 happy path por endpoint.
    - 1 cenário por regra de negócio listada nas Condições de Teste (camada API).
    - Cenários de autorização: 401 sem token, 403 quando o perfil é inadequado.
    - Pelo menos 1 cenário negativo por validação de input (campo obrigatório, formato, faixa).
- Boas práticas obrigatórias:
    - Nenhum teste depende da ordem de execução de outro.
    - Nenhum teste deixa estado residual que afete os próximos.
    - Nenhuma asserção fraca do tipo expect(res.status).to.be.ok.
    - Nomear specs no padrão "USxx — descrição comportamental" para facilitar leitura no relatório.
- Saída do prompt:
    - Estrutura de pastas criada conforme acima.
    - Arquivos .spec.js para todas as 15 User Stories cobrindo as Condições de Teste de camada API.
    - Helpers, fixtures e .mocharc.cjs prontos.
    - package.json atualizado com as devDependencies (mocha, chai, supertest, mochawesome, sinon, dotenv) e os scripts test, test:report e test:us.
    - README.md curto em test/README.md explicando como rodar e ler o relatório.

Exemplo
Estrutura mínima de um spec (referência, não copiar literal):

    // test/api/us01-login/login.spec.js
    const { expect } = require('chai');
    const request = require('supertest');
    const sinon = require('sinon');
    const http = require('../../helpers/http');
    const { loginAsClient } = require('../../helpers/auth');
    const invalidPayloads = require('../../fixtures/invalid-payloads.json');

    describe('US01 — Login de usuário', () => {
      describe('Validações', () => {
        invalidPayloads.login.forEach(({ description, payload, expectedMessage }) => {
          it(`CT01 — rejeita login: ${description}`, async () => {
            const res = await http.post('/auth/login').send(payload);
            expect(res.status).to.be.oneOf([400, 401]);
            expect(res.body).to.have.property('message', expectedMessage);
          });
        });
      });

      describe('Regras de negócio', () => {
        it('CT02 — autentica com credenciais válidas e retorna token JWT', async () => {
          const res = await loginAsClient();
          expect(res.status).to.equal(200);
          expect(res.body).to.have.property('token').that.is.a('string');
          expect(res.body).to.nested.include({ 'user.email': process.env.CLIENT_EMAIL });
        });

        it('CT04 — bloqueia o usuário após 5 tentativas inválidas consecutivas', async () => {
          const email = process.env.CLIENT_EMAIL;
          for (let i = 0; i < 5; i++) {
            await http.post('/auth/login').send({ email, password: 'errada' });
          }
          const res = await http.post('/auth/login').send({ email, password: 'errada' });
          expect(res.status).to.equal(423);
          expect(res.body.message).to.match(/bloqueado/i);
        });
      });
    });

Formato
- Suíte completa de testes Mocha/Chai/Supertest com Mochawesome configurado.
- Pasta test/ na raiz do backend, organizada por User Story conforme o item Estrutura.
- Mapeamento explícito CT ↔ it (cada Condição de Teste mencionada no nome do it).
- Comandos npm prontos para rodar localmente e gerar relatório HTML em test/reports/index.html.
